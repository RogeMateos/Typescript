Type Inference

Problem

Do I have to manually write the type of every variable? It seems repetitive and tedious.

Solution

TypeScript can infer (deduce) the type automatically based on the initial value. You don't always need to write it!

Visual Analogy

"If I give you an apple, you don't need to say 'this is a fruit'. I already know because I can see it. TypeScript works the same way: if you assign a number, it knows it's a number."

// TypeScript infers automatically
let age = 25; // TS knows it's 'number'
let name = "Anna"; // TS knows it's 'string'
let active = true; // TS knows it's 'boolean'

// age = "thirty"; // ‚ùå Error: you can't assign string
// You can also be explicit (optional)
let age: number = 25;
let name: string = "Anna";
let active: boolean = true;

// Both forms are valid, choose based on context

Best Practices

  • Let TypeScript infer when the type is obvious from the initial value
  • Use explicit annotations for function parameters and public APIs
  • Avoid writing redundant types: let x: number = 5 is unnecessary

Interview Prep

Q: What is type inference in TypeScript?

A: It is TypeScript's ability to automatically deduce the type of a variable based on its initial value, without requiring explicit annotations. It reduces repetitive code while maintaining type safety.

Getting Started with TypeScript

TypeScript is a typed superset of JavaScript that compiles to plain JavaScript. It adds optional static typing to help catch errors during development instead of at runtime. 

# Install TypeScript globally
npm install -g typescript

# Check version
tsc --version

# Create a TypeScript file
echo "let message: string = 'Hello TypeScript';" > index.ts

# Compile to JavaScript
tsc index.ts

Automatic Compilation with Watch Mode

Instead of manually compiling after each change, use --watch mode to automatically recompile on save. 

# Watch mode - recompiles on save
tsc index.ts --watch

# Or with the short flag
tsc index.ts -w

üí° Pro Tip: Keep the watch terminal open in one tab while you code in another. You'll see compilation errors instantly!

The Danger of "any" Type

The any type disables all type checking. It's like going back to plain JavaScript and losing all TypeScript benefits. 

// ‚ùå BAD: Disables type safety
let data: any = "hello";
data = 42;
data = { name: "John" };
data.nonExistentMethod(); // No error until runtime!

// ‚úÖ GOOD: Let TypeScript infer
let data = "hello"; // inferred as string
// data = 42; // Error: Type 'number' not assignable to 'string'

‚ö†Ô∏è Warning: Using any defeats the purpose of TypeScript. Only use it when migrating legacy code or dealing with truly dynamic data.

Enabling noImplicitAny in tsconfig.json

Force yourself to avoid implicit any by enabling noImplicitAny in your TypeScript configuration. 

{
  "compilerOptions": {
    "target": "ES2020",
    "module": "commonjs",
    "strict": true,              // Enables all strict checks
    "noImplicitAny": true,       // Errors on implicit 'any'
    "strictNullChecks": true,    // Prevents null/undefined bugs
    "esModuleInterop": true
  }
}

‚úÖ Best Practice: Always use "strict": true in new projects. It catches 90% of common bugs before they reach production.

Type Safety in Functions

Always annotate function parameters. TypeScript can infer return types, but parameters need explicit types. 

// ‚ùå BAD: Parameters have implicit 'any'
function add(a, b) {
  return a + b; // Could concatenate strings!
}

// ‚úÖ GOOD: Explicit parameter types
function add(a: number, b: number) {
  return a + b; // Return type inferred as 'number'
}

// ‚úÖ ALSO GOOD: Explicit return type (more documentation)
function add(a: number, b: number): number {
  return a + b;
}

What does : number at the end mean?

That : number after the parentheses is the return type. It means: "This function must return a number."

Think of it as a contract:

  • ‚Ä¢ The parameters a and b are numbers
  • ‚Ä¢ The result of the function is also a number

TypeScript can infer this automatically, so why write it explicitly?

1Ô∏è‚É£ Documentation (humans > compiler)

When someone reads function add(a: number, b: number): number, they immediately know: inputs ‚Üí numbers, output ‚Üí number. No need to read the function body.

2Ô∏è‚É£ Protection against bugs (the big one)

Imagine someone later edits the function: 

function add(a: number, b: number): number {
  return `${a + b}`; // üö® returns string!
}

TypeScript error: "Type 'string' is not assignable to type 'number'"

‚úÖ Bug caught before runtime!

Without the explicit return type, TypeScript would just infer string and you'd silently change the function's behavior.

3Ô∏è‚É£ Important for shared code

In real projects, helpers, utils, and services are used in many places. Explicit return types:

  • ‚Ä¢ Prevent breaking changes
  • ‚Ä¢ Make refactors safer
  • ‚Ä¢ Act like a public API

When should you add the return type?

‚úÖ Recommended (senior rule):

Add return types when the function is exported, reused, or part of a public API.

üòå Optional:

Omit for very small, local, obvious functions like const double = (n: number) => n * 2;

üí° One-sentence summary:

The : number at the end documents the function's output and protects you from accidentally changing what the function returns.

Understanding IIFE (Immediately Invoked Function Expression)

An IIFE is a function that runs immediately after being defined. Useful for encapsulation and avoiding global scope pollution. 

// Classic IIFE syntax
(function() {
  console.log("I run immediately!");
})();

// Arrow function IIFE
(() => {
  const secret = "Can't access me from outside";
  console.log(secret);
})();

// IIFE with parameters
((name: string) => {
  console.log(`Hello, ${name}!`);
})("TypeScript");

üìö Use Case: IIFEs are common in legacy code and script tags. Modern ES6 modules have mostly replaced their need for encapsulation.

Compiling All Files in a Project

Instead of compiling files one by one, use tsc without arguments to compile all files in your project. 

# Initialize tsconfig.json
tsc --init

# Compile all .ts files in the project
tsc

# Watch all files
tsc --watch

‚ö° Performance Tip: Use include and exclude in tsconfig.json to control which files are compiled.

Type Guards with unknown vs any

Use unknown instead of any when you don't know the type. It forces you to check the type before using it. 

// ‚ùå BAD: 'any' allows unsafe operations
function processValue(value: any) {
  return value.toUpperCase(); // No error, but crashes if not string
}

// ‚úÖ GOOD: 'unknown' requires type checking
function processValue(value: unknown) {
  if (typeof value === "string") {
    return value.toUpperCase(); // Safe!
  }
  return String(value);
}

Best Practices Summary

  • Let TypeScript infer variable types when obvious
  • Always type function parameters explicitly
  • Enable strict mode in tsconfig.json
  • Avoid any - use unknown with type guards instead
  • Use watch mode during development
  • Prefer const over let when values won't change

useState with Type Inference

React's useState infers types automatically from initial values. 

import { useState } from 'react';

function Counter() {
  // ‚úÖ Type inferred as number
  const [count, setCount] = useState(0);

  // ‚úÖ Type inferred as string
  const [name, setName] = useState("John");

  // ‚úÖ Explicit type for union or complex types
  const [user, setUser] = useState<User | null>(null);

  return <div>{count}</div>;
}

Typing Component Props

Always define prop types with interfaces or type aliases. 

// ‚úÖ Define props interface
interface ButtonProps {
  label: string;
  onClick: () => void;
  disabled?: boolean; // Optional prop
}

function Button({ label, onClick, disabled = false }: ButtonProps) {
  return (
    <button onClick={onClick} disabled={disabled}>
      {label}
    </button>
  );
}

Event Handlers in React

React provides specific types for different events. 

import { ChangeEvent, FormEvent } from 'react';

function Form() {
  const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
    console.log(e.target.value); // Type-safe!
  };

  const handleSubmit = (e: FormEvent<HTMLFormElement>) => {
    e.preventDefault();
    // Handle form submission
  };

  return (
    <form onSubmit={handleSubmit}>
      <input onChange={handleChange} />
    </form>
  );
}

Typing API Responses

Never use any for API data. Define interfaces for responses. 

interface User {
  id: number;
  name: string;
  email: string;
}

function UserList() {
  const [users, setUsers] = useState<User[]>([]);

  useEffect(() => {
    fetch('/api/users')
      .then(res => res.json())
      .then((data: User[]) => setUsers(data));
  }, []);

  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}

Avoiding any in React Functions

Always type function parameters and return values. 

// ‚ùå BAD: Using any
const formatUser = (user: any) => {
  return user.name.toUpperCase(); // Unsafe!
};

// ‚úÖ GOOD: Properly typed
interface User {
  name: string;
  age: number;
}

const formatUser = (user: User): string => {
  return user.name.toUpperCase(); // Type-safe!
};

Exporting and Importing Types

Organize types in separate files for reusability. 

// types/User.ts
export interface User {
  id: number;
  name: string;
  email: string;
}

// components/UserCard.tsx
import { User } from '../types/User';

interface UserCardProps {
  user: User;
}

export function UserCard({ user }: UserCardProps) {
  return <div>{user.name}</div>;
}

Complete React + TypeScript Example

A full component demonstrating best practices. 

import { useState, useEffect, ChangeEvent } from 'react';

interface Todo {
  id: number;
  text: string;
  completed: boolean;
}

export function TodoApp() {
  const [todos, setTodos] = useState<Todo[]>([]);
  const [input, setInput] = useState("");

  const handleAdd = () => {
    if (!input.trim()) return;

    const newTodo: Todo = {
      id: Date.now(),
      text: input,
      completed: false
    };

    setTodos([...todos, newTodo]);
    setInput("");
  };

  const handleToggle = (id: number) => {
    setTodos(todos.map(todo =>
      todo.id === id
        ? { ...todo, completed: !todo.completed }
        : todo
    ));
  };

  return (
    <div>
      <input
        value={input}
        onChange={(e: ChangeEvent<HTMLInputElement>) =>
          setInput(e.target.value)
        }
      />
      <button onClick={handleAdd}>Add</button>
      <ul>
        {todos.map(todo => (
          <li
            key={todo.id}
            onClick={() => handleToggle(todo.id)}
            style={{
              textDecoration: todo.completed ? 'line-through' : 'none'
            }}
          >
            {todo.text}
          </li>
        ))}
      </ul>
    </div>
  );
}
