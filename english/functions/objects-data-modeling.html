üì¶ TypeScript ‚Äì Objects and Data Modeling

Problem

Weak object modeling creates technical debt: duplicated shapes, fragile integrations, and inconsistent UI/API contracts.

Solution

Model objects with reusable, immutable, and explicit contracts, and separate API DTOs from UI models.

Level: Solid foundations with real React + API focus

Goal: Learn to model objects correctly and avoid future technical debt.

Interview Question

Q: Why should you separate DTO types from UI model types?

A: TypeScript only protects compile-time assumptions. API payloads can still be invalid at runtime, so you should validate/transform DTOs before using them in the UI.

‚úÖ Quick Checklist

  • ‚Ä¢ Avoid repeated inline object shapes.
  • ‚Ä¢ Use reusable type aliases.
  • ‚Ä¢ Prefer unknown over any.
  • ‚Ä¢ Separate API DTO and UI model.
  • ‚Ä¢ Use literal types for safe roles.
  • ‚Ä¢ Use readonly to prevent mutations.

8Ô∏è‚É£ Compile time vs runtime

type UserDto = {
  id: number;
  name: string;
};

type User = {
  id: string;
  name: string;
};

function mapUser(dto: UserDto): User {
  return {
    id: String(dto.id),
    name: dto.name
  };
}

Recommended flow: DTO -> validate -> map -> use in UI.

9Ô∏è‚É£ Discriminated unions

  role: "admin";
  permissions: string[];
};

type RegularUser = {
  role: "user";
};

type User = Admin | RegularUser;

function renderUser(user: User) {
  if (user.role === "admin") {
    return user.permissions.join(",");
  }
  return "Regular user";
}

üö® Common mistakes

  • ‚Ä¢ Repeating inline shapes.
  • ‚Ä¢ Using any.
  • ‚Ä¢ Mutating objects.
  • ‚Ä¢ Not separating DTO from UI model.
  • ‚Ä¢ Not using literal types for roles.
