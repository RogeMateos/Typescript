Interfaces: The Contract

Problem

We pass complex objects between functions. If a function expects 'user.id' and the object doesn't have it, the app fails.

Solution

Interfaces define the 'shape' of an object. They force the object to follow a specific structure.

Visual Analogy

"It's like a rental contract. It defines what the house must have (bathroom, kitchen, bed) to be valid for living."

interface User {
  name: string;
  age?: number; // Optional
}
const dev: User = {
  name: 'John',
  // age is optional, no error
};

Interview Prep

Q: Difference between Interface and Type?

A: Both define shapes, but interfaces are better for extensibility (they can merge) and types are more flexible for unions/aliases.

1 What is an Interface

An interface defines the contract that an object must follow.

interface User {
  name: string;
  age: number;
  readonly id: string;
  login(): boolean;
}

This means:

  • ‚úì The object must have name
  • ‚úì It must have age
  • ‚úì id cannot be modified
  • ‚úì It must implement a login method

2 What is a Type

A type can also define the shape of an object:

type User = {
  name: string;
  age: number;
  readonly id: string;
  login(): boolean;
};

And they do. In this case they work the same way.

So‚Ä¶ what's the actual difference?

3 The real differences (what matters)

üîπ Union Types (only with type)

With type you can do:

type UserOrNull = User | null;
type Role = "admin" | "user" | "guest";

üëâ You can't define this directly with interface.

üîπ Tuples (only with type)

type UserTuple = [string, number];
type UseToggle = [boolean, () => void];

Interfaces cannot define tuples.

üîπ Advanced types (mapped, conditional, template literals)

Only type allows things like:

type Nullable<T> = T | null;
type Id = `user-${number}`;

üîπ Declaration Merging (only interfaces)

If you declare an interface twice, they merge:

interface Hero {
  name: string;
}

interface Hero {
  power: string;
}
// Now Hero has both properties

‚ö†Ô∏è This can be dangerous if you don't know it's happening.

4 Interfaces and Classes (where they shine)

This is where interfaces feel natural:

interface MediaPlayer {
  play(): void;
  pause(): void;
  stop(): void;
}

class VideoPlayer implements MediaPlayer {
  play() {}
  pause() {}
  stop() {}
}
// If you don't implement a method ‚Üí TypeScript forces you

This is contract-oriented design.

With type you can't use implements.

5 Real cases in React + APIs

‚úÖ Case 1: API DTO (use interface)

interface ApiUserDto {
  id: string;
  name: string;
  email: string;
}

interface User {
  id: string;
  displayName: string;
}

const mapUser = (dto: ApiUserDto): User => ({
  id: dto.id,
  displayName: dto.name,
});

‚úÖ Case 2: Loading state (use type)

type UserState =
  | { status: "loading" }
  | { status: "success"; data: User[] }
  | { status: "error"; message: string };

const [state, setState] = useState<UserState>({
  status: "loading"
});

üëâ Type is clearly better here.

‚úÖ Case 3: Component Props

Both work. The decision is team convention.

interface ButtonProps {
  variant?: "primary" | "secondary";
  onClick: () => void;
}

6 Simple practical rule (to keep it uncomplicated)

‚úÖ Use Interface when:

  • ‚Ä¢ Defining domain models (User, Product, Order)
  • ‚Ä¢ Defining contracts for classes
  • ‚Ä¢ Need clear extension with extends
  • ‚Ä¢ Need declaration merging

‚úÖ Use Type when:

  • ‚Ä¢ Need unions
  • ‚Ä¢ Need tuples
  • ‚Ä¢ Need advanced types
  • ‚Ä¢ Creating generic utilities
  • ‚Ä¢ Modeling complex states
