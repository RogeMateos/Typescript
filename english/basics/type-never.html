Never Type: The Impossible

Problem

Some functions never return (throw errors or enter infinite loops). How to represent that something is impossible or will never happen?

Solution

The never type represents values that never occur. It's used for functions that throw errors, infinite loops, or exhaustive validations.

Visual Analogy

"Like a black hole: once you enter, there's no return. Never functions never go back to where they were called."

// Function that throws error (never returns)
function error(message: string): never {
  throw new Error(message);
}

// Infinite loop
function infiniteLoop(): never {
  while (true) {
    console.log("Running...");
  }
}

// TS infers never automatically
function fail() {
  throw new Error("Something went wrong");
  // TS knows this line is unreachable
  console.log("This never executes");
}
// Real use: Exhaustive validation
type Shape = "circle" | "square" | "triangle";

function area(shape: Shape, dimension: number): number {
  switch (shape) {
    case "circle":
      return Math.PI * dimension ** 2;
    case "square":
      return dimension ** 2;
    case "triangle":
      return (dimension ** 2) / 2;
    default:
      // If shape has a new value, TS gives error here
      const _exhaustive: never = shape;
      throw new Error(`Unhandled shape: ${shape}`);
  }
}

// never is the empty union type
type Impossible = string & number; // never

Best Practices

  • Use never in the default of switch for exhaustiveness checking
  • TypeScript infers never automatically in functions that only throw errors
  • never is useful to mark unreachable code or impossible cases
  • Don't confuse never with void: void returns, never doesn't

Interview Prep

Q: When to use never and what is its practical utility?

A: never is used in: 1) Functions that throw errors and never return, 2) Infinite loops, 3) Exhaustive validation in switch statements (if you add a new case to the type and forget to handle it, TS gives an error). It's the "bottom" type that represents impossible values or unreachable code.

1 What is never in TypeScript?

The never type represents values that never occur.

In other words, it indicates that:

  • ‚Ä¢ A function never finishes normally
  • ‚Ä¢ The execution flow cannot continue beyond that point

This happens when a function:

  • ‚Ä¢ Always throws an error (throw)
  • ‚Ä¢ Or enters an infinite loop

üëâ never doesn't mean "returns nothing", but "there's no way it can return something".

2 never vs void (key to not getting confused)

This comparison is fundamental.

const logMessage = (msg: string): void => {
  console.log(msg);
  // finishes normally
};

const crash = (msg: string): never => {
  throw new Error(msg);
  // never finishes
};
Type Meaning
void The function finishes, but doesn't return a useful value
never The function cannot finish

üëâ A void function does return.

üëâ A never function never returns.

3 Basic example of never 

(() => {
  const error = (message: string): never => {
    throw new Error(message);
  };

  error("help");

  // ‚ùå This code never executes
  console.log("Hello world");
})();

What's happening here

  • ‚Ä¢ error always throws an error
  • ‚Ä¢ TypeScript knows the flow is cut
  • ‚Ä¢ Everything that comes after is unreachable code

This allows TypeScript to do flow analysis and detect errors before executing the code.

4 Why does TypeScript complain if you don't fulfill never?

If you mark a function as never, you must guarantee it never reaches the end.

‚ùå Incorrect:

const maybeFail = (flag: boolean): never => {
  if (flag) {
    throw new Error("Boom");
  }
  // here the function could finish
};

TypeScript complains because:

  • ‚Ä¢ There's a path where there's no throw
  • ‚Ä¢ Therefore, it could finish

‚úÖ Correct:

const alwaysFail = (): never => {
  throw new Error("Always fails");
};

5 Important: never | number is NOT a good idea

Sometimes you see this:

const errorOrNumber = (msg: string): never | number => {
  if (Math.random() > 0.5) {
    throw new Error(msg);
  }
  return 1;
};

‚ö†Ô∏è In practice:

  • ‚Ä¢ never | number simplifies to number
  • ‚Ä¢ never doesn't add useful information here

‚úÖ Better:

const errorOrNumber = (msg: string): number => {
  if (Math.random() > 0.5) {
    throw new Error(msg);
  }
  return 1;
};

üëâ The throw is part of the flow, not the return type.

6 never in its most powerful use: exhaustiveness

This is where never really shines.

assertNever function

const assertNever = (value: never): never => {
  throw new Error("Unhandled case: " + JSON.stringify(value));
};

This function is used to ensure all possible cases are covered.

7 Exhaustiveness with switch 

type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; size: number };

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.size ** 2;
    default:
      return assertNever(shape);
  }
}

Why is this so valuable?

If tomorrow someone adds:

{ kind: "triangle"; base: number; height: number }
  • ‚Ä¢ TypeScript will fail at compilation
  • ‚Ä¢ You avoid silent bugs in production

This is real safety, not theory.

Real example in React + TypeScript

Exhaustive reducer

type Action =
  | { type: "increment" }
  | { type: "decrement" }
  | { type: "set"; value: number };

const assertNever = (x: never): never => {
  throw new Error("Unsupported action");
};

function reducer(state: number, action: Action): number {
  switch (action.type) {
    case "increment":
      return state + 1;
    case "decrement":
      return state - 1;
    case "set":
      return action.value;
    default:
      return assertNever(action);
  }
}

üëâ If someone adds a new action and doesn't update the reducer, TypeScript detects it.

8 Impossible states in components 

type Props =
  | { status: "loading" }
  | { status: "ready"; data: string[] }
  | { status: "error"; message: string };

function List(props: Props) {
  switch (props.status) {
    case "loading":
      return "Loading...";
    case "ready":
      return props.data.join(", ");
    case "error":
      return props.message;
    default: {
      const _exhaustive: never = props;
      return _exhaustive;
    }
  }
}

This prevents invalid prop combinations.

Best practices

‚úÖ Use never when:

  • ‚Ä¢ A function always throws an error
  • ‚Ä¢ You want total exhaustiveness in switch
  • ‚Ä¢ You want to model impossible states

‚ùå Avoid never when:

  • ‚Ä¢ The function can finish
  • ‚Ä¢ You use it as "decoration"
  • ‚Ä¢ You mix it in unions unnecessarily
