Tuples: Arrays with Fixed Structure

Problem

Arrays allow any number of elements of the same type, but sometimes you need a fixed number of elements with specific types in each position.

Solution

Tuples are arrays with a fixed number of elements where each position has a specific predefined type.

Visual Analogy

"Like GPS coordinates: they're always two numbers in fixed order [latitude, longitude]. You can't swap them or add a third value without changing their meaning."

// Defining a Tuple
let user: [string, number, boolean];

// Correct assignment
user = ["Anna", 25, true];

// ‚ùå Error: order matters
// user = [25, "Anna", true];

// Access by index (with correct types)
let name: string = user[0];   // ‚úÖ OK
let age: number = user[1];    // ‚úÖ OK
let active: boolean = user[2]; // ‚úÖ OK
// Real use case: API Response
type ApiResponse = [number, string, any];

function fetchData(): ApiResponse {
  return [200, "OK", { data: "..." }];
}

const [code, message, data] = fetchData();
// code is number
// message is string
// data is any

// Tuple with labels (TS 4.0+)
type Coordinates = [lat: number, lng: number];
let position: Coordinates = [40.7128, -74.0060];

Best Practices

  • Use tuples for related values with fixed order (like coordinates or RGB)
  • Limit tuples to 2-3 elements; for more, use interfaces
  • Use readonly for immutable tuples: readonly [string, number]
  • Add labels for clarity: [name: string, age: number]

Interview Prep

Q: What is the difference between a Tuple and an Array in TypeScript?

A: An Array has variable length and all elements are the same type (number[]). A Tuple has fixed length and each position can have a different type ([string, number]). Tuples are ideal when the order and number of elements have semantic meaning.

1 What is a tuple?

A tuple in TypeScript is a data structure that lets you define an array with a fixed number of elements, where each position has a specific type.

üëâ Key idea to understand them well

  • Use a tuple when order matters and each position has concrete meaning.
  • It's not just any list; it's a small data contract.

2 Difference between tuples and arrays

üì¶ Array (homogeneous list)

const numbers: number[] = [1, 2, 3];
  • ‚úì All elements are the same type
  • ‚úì Order usually has no semantic meaning

üìç Tuple (positional structure)

const hero: [string, number] = ["Doctor Strange", 100];
  • ‚úì Each position has a type and meaning
  • ‚úì Order is required and relevant

3 Basic example explained step by step 

const hero: [string, number] = ["Doctor Strange", 100];

console.log(hero[0]); // "Doctor Strange"
console.log(hero[1]); // 100

‚úî Here TypeScript understands that:

  • hero[0] ‚Üí will always be string
  • hero[1] ‚Üí will always be number
  • This is not a suggestion, it's a type rule.

4 Type safety (why they're useful) 

hero[0] = 50;

‚ùå TypeScript Error:

Type 'number' is not assignable to type 'string'

üëâ This error appears before running the code

This prevents bugs in production. In JavaScript, this would pass silently.

5 Important: at runtime they're still arrays

Although TypeScript treats them as tuples, in JavaScript they're normal arrays.

const hero: [string, number] = ["Doctor Strange", 100];

‚¨áÔ∏è Compiles to:

["Doctor Strange", 100]

üìå Conclusion:

  • ‚úì TypeScript protects at development time
  • ‚úì At runtime, tuples don't exist as a real structure

6 Common trap: push breaks the tuple idea 

const hero: [string, number] = ["Doctor Strange", 100];

hero.push(999);
console.log(hero); // ["Doctor Strange", 100, 999]

‚ö†Ô∏è This might not error depending on configuration.

Why does it happen?

Because a tuple is also an array, and inherits its methods.

7 Recommended solution: readonly tuples 

const hero: readonly [string, number] = ["Doctor Strange", 100];

// hero.push(1); ‚ùå Error

‚úîÔ∏è Now:

  • ‚úì Can't be modified
  • ‚úì The tuple contract is respected

8 Readability improvement: naming positions

You can name each element of the tuple:

type HeroTuple = [name: string, power: number];

const hero: HeroTuple = ["Iron Man", 150];

‚úî This:

  • ‚úì Makes code more readable
  • ‚úì Serves as documentation
  • ‚úì Helps a lot in large teams

Here:

  • heroData[0] ‚Üí name
  • heroData[1] ‚Üí power
  • heroData[2] ‚Üí active or not

üìå Practical rule:

  • 2‚Äì3 elements ‚Üí tuple üëç
  • More than 3 ‚Üí better use an object

üîü When NOT to use tuples

Not a good idea when:

  • ‚úó There are many fields
  • ‚úó The meaning is not obvious
  • ‚úó You use many indices (data[4], data[5]‚Ä¶)

Better example with object:

  name: string;
  power: number;
  active: boolean;
};

1Ô∏è‚É£1Ô∏è‚É£ Real case: Result pattern (widely used in APIs)

Pattern definition:

type Ok<T> = [ok: true, data: T];
type Err = [ok: false, error: string];
type Result<T> = Ok<T> | Err;

function parseNumber(value: string): Result<number> {
  const n = Number(value);
  if (Number.isNaN(n)) return [false, "Not a number"];
  return [true, n];
}

Usage:

const result = parseNumber("42");

if (result[0]) {
  console.log(result[1]); // number
} else {
  console.log(result[1]); // error string
}

üëâ The first position controls the program flow.

This is a very powerful pattern for error handling without exceptions.
