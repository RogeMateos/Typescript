TypeScript vs JavaScript

The key insight: TypeScript is a development-time guardian, not a runtime one. It protects you during development, but the code that runs in production is still JavaScript.

All TypeScript types disappear after compilation. Understanding this changes how you write secure, production-ready code.

1 The Real Problem in JavaScript

JavaScript doesn't check types before executing. This leads to surprising behavior:

function double(n) {
  return n * 2;
}

double(2);    // 4 ‚úÖ
double("2");  // 4 üòÖ (implicit coercion!)
double("a");  // NaN üò¨ (silent failure)

What's happening: JavaScript performs implicit type conversions. "2" becomes 2, "a" becomes NaN. The code runs without errors, but produces wrong results.

2 Enter TypeScript: Static Typing

Now we declare a contract:

function double(n: number) {
  return n * 2;
}

double(2);    // ‚úÖ OK
double("2");  // ‚ùå Compile-time error!

The compiler checks types before execution. Errors appear in your editor, not in production.

CRITICAL: TypeScript Doesn't Exist at Runtime

This is the most important thing to understand:

  • ‚Ä¢ TypeScript only acts during development - it's like a spell-checker.
  • ‚Ä¢ Production runs pure JavaScript - all types are stripped away.
  • ‚Ä¢ If this code somehow runs as JavaScript, it will behave exactly like the original.

"TypeScript is an inspector before departure, not a bodyguard after arrival."

3 The Real World: External Data

In real applications, data comes from untrusted sources:

  • ‚Ä¢User inputs
  • ‚Ä¢API responses
  • ‚Ä¢localStorage data
  • ‚Ä¢Environment variables

‚ùå Common mistake: Assuming that because TypeScript says something is true, it will be true in runtime. Wrong! TypeScript only knows what you told it.

4 unknown vs any: This Makes You a Better Developer

‚ùå Using "any"

const x: any = "2";
x.toFixed(2); // üí• Runtime error!
// TypeScript doesn't help

You lose all type safety. Essentially the same as JavaScript.

‚úÖ Using "unknown"

const x: unknown = "2";
if (typeof x === "number") {
  x.toFixed(2); // ‚úÖ Safe!
}

Forced to validate before use. Real type safety.

Proper Validation Pattern

function toNumber(value: unknown): number {
  if (typeof value === "number" && Number.isFinite(value)) {
    return value;
  }

  if (typeof value === "string") {
    const parsed = Number(value);
    if (Number.isFinite(parsed)) {
      return parsed;
    }
  }

  throw new Error("Invalid number input");
}

This validates thoroughly because Number("anything") can return NaN, and NaN is still technically a number!

Senior Developer Checklist

  • ‚úÖ Treat all external data as unknown
  • ‚úÖ Validate before using data in your app
  • ‚úÖ Separate DTO (API response) from Model (app state)
  • ‚ùå Never use as Type without validation
  • ‚ùå Don't use any unless absolutely necessary
