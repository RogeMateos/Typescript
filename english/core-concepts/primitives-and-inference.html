TypeScript Fundamentals: Primitives & Inference

Problem

Doesn't TypeScript require writing types everywhere? How can I understand const vs let differences with types? When do null and undefined matter?

Solution

TypeScript infers types automatically. Understand primitive types, const literal types, and union types to write safer, more maintainable code with confidence.

Visual Analogy

"If I give you an apple, you don't say 'this is a fruit'. You already know because you can see it. TypeScript works the same way: it reads your values and deduces the types automatically."

Interview Prep

Q: What's the difference between const and let in TypeScript type inference?

A: const infers literal types (exact values like "primary"), while let widens types to general categories (like string). This difference matters deeply in React props and union types.

Type Inference: TypeScript Reads Your Mind

TypeScript deduces types from initial values automatically. 

const name = "MoureDev";

Hover over name and see:

const name: "MoureDev"

üîé Important: The type is NOT string, but the literal "MoureDev". This distinction is CRITICAL!

Compare with let:

let name = "MoureDev";

TypeScript infers:

let name: string

üéØ Why the difference?

Because let can be reassigned:

let name = "MoureDev";
name = "Batman";    // ‚úÖ Allowed
name = "Superman";  // ‚úÖ Allowed

Since the value can change to ANY string, TypeScript must allow all strings, not just "MoureDev".

Inference + Autocomplete = The Magic

TypeScript knows the types, so it provides intelligent autocomplete:

greeting.toUpperCase(); // ‚úÖ VS Code suggests string methods

‚ú® Not magic. Static analysis.

TypeScript analyzed your code and determined types. That's why autocomplete works perfectly.

The Seven Primitive Types

1. String

const city = "Madrid";
const message = `Hello, ${city}!`;
const empty = "";

2. Number

const age = 30;              // Integer
const price = 19.99;         // Float
const hexColor = 0xfff;      // Hex
const infinite = Infinity;   // Special

3. Boolean

const isActive = true;
const isLoading = false;

4. Null vs Undefined (CRITICAL)

They are NOT the same!

Type Meaning
undefined Not assigned yet
null Intentionally empty

Why it matters:

null from API means "no data". undefined in a form means "user hasn't touched it". Different meaning!

6. BigInt | 7. Symbol

const big: bigint = 9007199254740993n;
const id: symbol = Symbol("id");

Advanced, rarely used in typical React.

Union Types: Multiple Possible Types

A variable can be multiple types: 

let age: number | null = null;

age = 30;   // ‚úÖ OK
age = null; // ‚úÖ OK
age = "30"; // ‚ùå Error

Use cases:

  • ‚Ä¢ Form fields (value or empty)
  • ‚Ä¢ API responses (data or null)
  • ‚Ä¢ React state (different states)

Type Narrowing: Refining Union Types

Narrow union types with conditionals: 

function printAge(age: number | null) {
  if (age === null) {
    console.log("Not provided");
    return;
  }

  // TypeScript KNOWS age is number here
  console.log(age.toFixed(0)); // ‚úÖ Safe
}

‚≠ê const vs let: The Critical Difference

This is the #1 reason for "not assignable to" errors in React.

The Difference

const variant = "primary";
// Type: "primary" (literal - EXACT value only)

let variant = "primary";
// Type: string (general - ANY string allowed)

Why This Happens

const

Cannot be reassigned

‚Üí Locks to exact value

let

Can be reassigned

‚Üí Must allow any value

Why This Breaks in React 

type ButtonProps = {
  variant: "primary" | "secondary";
};

‚úÖ Works:

const variant = "primary";
<Button variant={variant} />

‚ùå Fails:

let variant = "primary";
<Button variant={variant} />
// Error: string not assignable

üîë The Key Insight

It's not about the current value, it's about what it COULD become.

  • ‚Ä¢ const variant = "primary" ‚Üí always "primary"
  • ‚Ä¢ let variant = "primary" ‚Üí could be "danger", "hello", anything!

How to Fix

Option 1: Type the variable

let variant: "primary" | "secondary" = "primary";

Option 2: Use as const

const config = { variant: "primary" } as const;

Option 3: Prefer const

const variant = "primary"; // ‚úÖ Works!

Real React Integration

Case 1: Discriminated Unions with useState

type State =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: User[] }
  | { status: "error"; message: string };

const [state, setState] = useState<State>({ status: "idle" });

Case 2: null vs undefined in Forms

type Profile = { age: number | null };  // Backend
type FormState = { age?: number };       // Frontend

Why:

  • ‚Ä¢ null = backend has no value
  • ‚Ä¢ undefined = user hasn't touched field
