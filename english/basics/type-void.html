Void Type: Functions without Return

Problem

Some functions don't return a useful value (like functions that print to console or modify the DOM). How to indicate this in TypeScript?

Solution

The void type indicates that a function doesn't return any value. It's TypeScript's way of saying "this function does something, but don't expect a result."

Visual Analogy

"Like shouting in a valley: you perform an action (shout) but don't expect to receive something tangible back, just the echo you won't use."

// Function with explicit void
function greet(name: string): void {
  console.log(`Hello, ${name}`);
  // No return (or return without value)
}

// Function that modifies the DOM
function updateTitle(title: string): void {
  document.title = title;
}

// ‚ùå Error: void cannot be assigned to another type
let result: string = greet("Anna");
// TypeScript infers void automatically
function log(message: string) {
  console.log(message);
  // TS infers: ': void'
}

// Difference with undefined
function returnsUndefined(): undefined {
  return undefined; // Must explicitly return
}

function returnsVoid(): void {
  return; // Can have no return
}

// Callbacks with void
type Callback = (data: string) => void;
const process: Callback = (data) => {
  console.log(data);
};

Best Practices

  • Use void for functions that perform side effects (logs, DOM, API calls)
  • TypeScript infers void automatically, but it's good to be explicit in public APIs
  • Don't confuse void with undefined: void is more flexible in callbacks
  • In callbacks, void allows ignoring the callback's return value

Interview Prep

Q: What is the difference between void and undefined in TypeScript?

A: void means the return value is ignored or shouldn't be used (the function may have no return). undefined is a specific type and value that must be returned explicitly. In callbacks, void is more flexible: it allows the callback to return anything, but that value will be ignored.

1 Why does void exist in TypeScript?

Let's start with pure JavaScript.

In JavaScript, all functions return something, even if you don't see it:

function sayHello() {
  console.log("Hello");
}

const result = sayHello();
console.log(result); // undefined

üëâ If you don't use return, JavaScript returns undefined automatically.

So‚Ä¶ why does TypeScript need void?

Because TypeScript doesn't just describe what happens at runtime, but how the code should be used.

‚úî void exists to express a clear intention:

"This function exists to do something, not to return a value that should be used."

2 What does void really mean?

‚ùå Wrong idea:

void means the function doesn't return anything

It means something more precise:

üîπ The return value should not be used

Basic example:

function callBatman(): void {
  console.log("Bat-signal activated ü¶á");
}
  • ‚úî The function does something
  • ‚úî We don't expect any result
  • ‚úî It makes no sense to use what it "returns"

3 void functions and return

‚úÖ Allowed: not returning anything

function logMessage(message: string): void {
  console.log(message);
}

‚úÖ Allowed: return; (early exit)

function validate(isValid: boolean): void {
  if (!isValid) return; // just stops execution
  console.log("All correct");
}

‚ùå NOT allowed: returning a value

function logMessage(message: string): void {
  return message; // ‚ùå Error
}

üí° Reason:

void tells TypeScript that no value should exit from here.

4 void vs undefined vs null (this confuses many)

Here's the clear difference, without confusion:

üîπ void

  • ‚úì It's a type
  • ‚úì Used mainly in function returns
  • ‚úì Expresses "no useful value"
function doSomething(): void {}

üîπ undefined

  • ‚úì It's a real value in JavaScript
  • ‚úì Means "not assigned"
let x: undefined = undefined;

üîπ null

  • ‚úì It's an intentional value
  • ‚úì Means "empty on purpose"
let y: null = null;

üëâ Senior key:

void is about intent of use, undefined and null are data.

5 void in arrow functions

Exactly the same:

const callSuperman = (): void => {
  console.log("Up, up and away ü¶∏‚Äç‚ôÇÔ∏è");
};

‚úî Very common in callbacks and handlers.

6 Real case: void in React (very important)

In React, most handlers don't return anything useful.

function Button() {
  const handleClick = (): void => {
    console.log("Clicked");
  };

  return <button onClick={handleClick}>Click</button>;
}

üí° Correct mentality:

  • ‚úì If you need a result ‚Üí use state, props or callback
  • ‚úì Don't return values from handlers

7 Promise<void> (key in async / APIs)

This point is often missing from basic tutorials, but it's CRITICAL.

async function saveUser(): Promise<void> {
  await fetch("/api/user", { method: "POST" });
}

‚úî What does it mean?

  • ‚úì The function is async
  • ‚úì Does something (side-effect)
  • ‚úì Doesn't return data

Very common in:

  • ‚úì Saving data
  • ‚úì Sending logs
  • ‚úì Firing events

8 Key difference: void vs never (pro level)

This elevates your tutorial üëá

üîπ void

The function ends, but doesn't return useful value.

function log(msg: string): void {
  console.log(msg);
}

üîπ never

The function NEVER ends.

function crash(): never {
  throw new Error("Boom");
}

üëâ If a function always throws error or enters infinite loop, it's NOT void, it's never.

9 Best practices (explained like senior)

‚úÖ Use void when:

  • ‚úì Function only produces effects
  • ‚úì It's a handler, callback or logger
  • ‚úì You don't want anyone to use the return

‚ùå Don't use void if:

  • ‚úì Function returns data
  • ‚úì The result is used later
  • ‚úì It's part of a calculation

üëâ void documents intent and prevents silent bugs.

‚öõÔ∏è void in React + TypeScript (Practical Guide)

Context: In React, most functions don't exist to return values, but to cause effects: update state, launch a request, log something, etc.

That's where void fits perfectly.

1 Base idea in React: "don't return data, change state"

In React we don't use function returns to control the UI.

‚ùå Wrong mentality (classic JS):
function isValid(): boolean {
  return true;
}
‚úÖ React mentality:
function validate(): void {
  setIsValid(true);
}

üëâ In React:

  • ‚úì UI responds to state
  • ‚úì Not to function return values
  • ‚úì That's why void is so common

2 Event handlers in React are always void

Typical example:

function Button() {
  const handleClick = (): void => {
    console.log("Click");
  };

  return <button onClick={handleClick}>Click</button>;
}

Why void here?

  • ‚úì React completely ignores handler return
  • ‚úì If you return something, nothing useful happens
  • ‚úì void makes the intent clear

üí° Senior tip:

If a function is passed to JSX as an event, its return doesn't matter ‚Üí void

3 Handlers with typed event 

function Button() {
  const handleClick = (
    event: React.MouseEvent<HTMLButtonElement>
  ): void => {
    console.log(event.currentTarget);
  };

  return <button onClick={handleClick}>Click</button>;
}
  • ‚úî Type the event
  • ‚úî Type the return
  • ‚úî Self-documenting code

4 void + useState (most common pattern)

Real example:

function Counter() {
  const [count, setCount] = React.useState(0);

  const increment = (): void => {
    setCount(prev => prev + 1);
  };

  return <button onClick={increment}>{count}</button>;
}

Why void?

  • ‚úì increment doesn't return the new value
  • ‚úì It only causes a state change
  • ‚úì React handles the re-render

üëâ This is pure React.

5 void in async functions (Promise<void>)

This point is KEY and often confuses.

Realistic example (saving data):

async function saveUser(): Promise<void> {
  await fetch("/api/user", { method: "POST" });
}

Usage in React:

function SaveButton() {
  const [saving, setSaving] = React.useState(false);

  const handleSave = async (): Promise<void> => {
    setSaving(true);
    await saveUser();
    setSaving(false);
  };

  return <button disabled={saving}>Save</button>;
}

Senior mentality:

  • ‚úì If you don't return data ‚Üí Promise<void>
  • ‚úì The result is reflected in state, not in return

6 useEffect: why it's ALWAYS void

This is fundamental.

React.useEffect((): void => {
  console.log("Component mounted");
}, []);

Why?

useEffect doesn't expect a value

It only allows:

  • ‚úì nothing (void)
  • ‚úì or a cleanup function
‚ùå Wrong:
useEffect(() => {
  return 123; // ‚ùå
});
‚úÖ Correct:
useEffect(() => {
  return () => {
    console.log("cleanup");
  };
});

7 Callbacks and props: () => void

Very common when passing functions between components.

type Props = {
  onClose: () => void;
};

function Modal({ onClose }: Props) {
  return <button onClick={onClose}>Close</button>;
}

Usage:

<Modal onClose={() => setOpen(false)} />

Justification:

  • ‚úì Component doesn't need to know anything
  • ‚úì Just executes the action
  • ‚úì void expresses independence

8 Common junior mistake (and why void prevents it)

‚ùå Anti-pattern:
const handleSubmit = (): boolean => {
  setLoading(true);
  return true;
};

Why is it wrong?

  • ‚úì React ignores the return
  • ‚úì The boolean serves no purpose
  • ‚úì Creates false sense of control
‚úÖ Correct:
const handleSubmit = (): void => {
  setLoading(true);
};

9 void vs never in React

Important to distinguish.

void ‚Üí normal action

const logOut = (): void => {
  setUser(null);
};

never ‚Üí app breaks or stops

function fatalError(message: string): never {
  throw new Error(message);
}

In React:

  • ‚úì void = normal flow
  • ‚úì never = crash

üß† Think like this:

  • ‚úì Event handler ‚Üí void
  • ‚úì Prop callback ‚Üí void
  • ‚úì useEffect ‚Üí void
  • ‚úì Async without data ‚Üí Promise<void>
  • ‚úì If you need data ‚Üí state / props, not return

üß© Final summary (for React)

In React, void is the natural type for almost all functions. Components don't depend on function return values, but on side effects that update state.

Using void makes this intent explicit, improves readability, and prevents design errors inherited from classic JavaScript.

  • Not the same as undefined
  • Key in React and async
  • Helps express intent
