Default Parameters

Problem

Repeating the same value every time you call a function is tedious and error-prone.

Solution

Assign a default value to parameters using =.

Visual Analogy

"Like a form with pre-filled values. You can change them if you want, or leave them as they are."

function greet(name: string, language: string = "en"): string {
  const greetings = { en: "Hello", es: "Hola", fr: "Bonjour" };
  return `${greetings[language]}, ${name}`;
}

greet("John"); // "Hello, John" (uses default)
greet("Juan", "es"); // "Hola, Juan" (overrides)

Interview Question

Q: Can you override a parameter with a default value?

A: Yes, absolutely. You can pass any value to replace the default.

1 Basic syntax 

function createProduct(
  name: string,
  price: number,
  discount: number = 0
): number {
  return price * (1 - discount / 100);
}

üìå The discount parameter has a default value of 0.

2 Literal values as defaults

  recipient: string,
  subject: string = "No subject",
  priority: string = "normal"
): void {
  console.log(`To: ${recipient}, Subject: ${subject}, Priority: ${priority}`);
}

3 Complex default values 

function configureOptions(
  url: string,
  options: object = { timeout: 5000, retry: 3 }
): void {
  console.log(url, options);
}

‚ö†Ô∏è Parameters with default values must also go AFTER required parameters (optional).

Difference: Default vs Optional

‚úÖ With default value:

parameter: type = value ‚Üí always has a value

‚ùì Optional:

parameter?: type ‚Üí can be undefined

1 The problem TypeScript wants to avoid

In TypeScript, parameter order matters when mixing:

  • ‚Ä¢ Required parameters
  • ‚Ä¢ Optional parameters (?)
  • ‚Ä¢ Parameters with default values (=)

Problematic example:

function fullName(
  firstName: string,
  lastName?: string,
  capitalize: boolean
) {}

// ‚ùå Error: A required parameter cannot follow an optional parameter.

Why? TypeScript can't know if true is lastName or capitalize.

2 Golden rule (very important)

Required parameters MUST always come before optional ones.

Correct order:

  1. 1. Required
  2. 2. Optional (?)
  3. 3. Parameters with default values (=)

3 Optional parameter (?) vs default value (=)

Optional parameter (?):

function greet(name: string, surname?: string) {}

// May or may not come
// Its type is: string | undefined
// You must check it inside the function

Parameter with default value (=):

function greet(name: string, surname = "") {}

// If not provided, takes the default value
// Never is undefined
// The type stays clean (string)

üëâ In practice: Default values are usually more convenient.

4 Solution 1: Reorder parameters 

function fullName(
  firstName: string,
  capitalize: boolean,
  lastName?: string
) {
  const full = lastName
    ? `${firstName} ${lastName}`
    : firstName;

  return capitalize ? full.toUpperCase() : full;
}

‚úÖ Correct but not scalable if options start to grow

5 Solution 2: Use default values (highly recommended) 

function fullName(
  firstName: string,
  lastName: string = "",
  capitalize: boolean = false
) {
  const full = lastName
    ? `${firstName} ${lastName}`
    : firstName;

  return capitalize ? full.toUpperCase() : full;
}

// Usage:
fullName("John");
fullName("John", "Doe");
fullName("John", "Doe", true);

‚úÖ Key advantage: You don't have to deal with undefined

6 The professional pattern: options object

When a function starts to have flags, configurations, or options, use this pattern:

type FullNameOptions = {
  lastName?: string;
  capitalize?: boolean;
  separator?: string;
};

function fullName(
  firstName: string,
  options: FullNameOptions = {}
) {
  const {
    lastName,
    capitalize = false,
    separator = " "
  } = options;

  const full = lastName
    ? `${firstName}${separator}${lastName}`
    : firstName;

  return capitalize ? full.toUpperCase() : full;
}

‚úÖ Scalable ‚Ä¢ ‚úÖ Readable ‚Ä¢ ‚úÖ Common in React and APIs

7 Real example: API helper (React/Frontend) 

type FetchOptions = {
  method?: "GET" | "POST";
  authToken?: string;
  timeoutMs?: number;
};

async function apiFetch<T>(
  url: string,
  options: FetchOptions = {}
): Promise<T> {
  const {
    method = "GET",
    authToken,
    timeoutMs = 10000
  } = options;

  const headers = authToken
    ? { Authorization: `Bearer ${authToken}` }
    : {};

  const controller = new AbortController();
  setTimeout(() => controller.abort(), timeoutMs);

  const response = await fetch(url, {
    method,
    headers,
    signal: controller.signal
  });

  if (!response.ok) {
    throw new Error("API error");
  }

  return response.json();
}

Important pitfalls (senior level)

‚ùó undefined activates the default, null does NOT:

  return x;
}

demo(undefined); // 10 ‚úÖ
demo(null);      // null ‚ùå

‚ö†Ô∏è Very important when consuming APIs.

‚ùó Too many positional parameters is a red flag:

// ‚ùå Red flag:
func(a, b, c, d, e);

// ‚úÖ Better:
func(a, { b, c, d, e });
