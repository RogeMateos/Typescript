Null vs Undefined

Problem

JavaScript has two values for "nothing": null and undefined. This causes confusion and errors like "Cannot read property of undefined".

Solution

TypeScript distinguishes between null and undefined, allowing you to control them with strictNullChecks and operators like optional chaining (?.) and nullish coalescing (??).

Visual Analogy

"undefined is a box that was never opened. null is an opened box that is intentionally empty."

// undefined: variable declared but not initialized
let name: string;
console.log(name); // undefined

// null: intentional absence of value
let age: number | null = null;

// With strictNullChecks enabled
let city: string = "Madrid";
// city = null; // ‚ùå Error: null is not string

// Explicitly allow null
let country: string | null = null; // ‚úÖ OK
country = "Spain"; // ‚úÖ OK
// Optional Chaining (?.)
const user = { name: "Anna", address: undefined };
const street = user.address?.street; // undefined (no error)

// Nullish Coalescing (??)
const port = process.env.PORT ?? 3000;
// Uses 3000 only if PORT is null/undefined (not if it's 0 or "")

// Non-null Assertion (!)
function process(value: string | null) {
  // If you're 100% sure it's not null
  console.log(value!.toUpperCase());
}

// Comparison
console.log(null == undefined);   // true (loose comparison)
console.log(null === undefined);  // false (strict comparison)

Best Practices

  • Enable strictNullChecks in tsconfig.json for better safety
  • Prefer undefined for optional values (TypeScript does this by default)
  • Use null only when you want to indicate intentional absence
  • Use ?. (optional chaining) and ?? (nullish coalescing) for cleaner code
  • Avoid ! (non-null assertion) unless you're completely sure

Interview Prep

Q: What is the difference between null and undefined, and when to use each?

A: undefined means a variable was declared but not initialized (default value). null is an intentional assignment representing "empty". In TypeScript, with strictNullChecks, you must explicitly declare if a type can be null/undefined. Use undefined for optional parameters and null when you want to explicitly indicate "no value".

1 What are null and undefined?

In JavaScript and TypeScript, null and undefined represent "absence of value", but with different intentions:

undefined: Default value when a variable exists but hasn't been initialized.

null: Intentional assignment indicating "this value is explicitly empty".

let name: string;
console.log(name); // undefined (declared but not initialized)

let age: number | null = null; // Explicitly empty

// In React:
const [user, setUser] = useState<User | null>(null); // No user yet
const [data, setData] = useState<Data>(); // undefined by default

2 strictNullChecks: Your Best Friend in TypeScript

strictNullChecks is an option in tsconfig.json that forces you to explicitly declare when a value can be null or undefined.

Without strictNullChecks: You can assign null/undefined to anything (dangerous).

With strictNullChecks: TypeScript forces you to handle null/undefined explicitly.

// tsconfig.json
{
  "compilerOptions": {
    "strictNullChecks": true
  }
}

// Without strictNullChecks (‚ùå bad)
let name: string = "Anna";
name = null; // No error (but should)

// With strictNullChecks (‚úÖ good)
let name: string = "Anna";
// name = null; // ‚ùå Error: Type 'null' is not assignable to type 'string'

let optionalName: string | null = "Anna";
optionalName = null; // ‚úÖ OK

3 React Props That Can Be Null

In React, it's common to have props that may not exist initially (e.g., API data).

React + TypeScript

interface UserCardProps {
  user: User | null; // User can be null
}

function UserCard({ user }: UserCardProps) {
  // ‚ùå Bad: Don't check if user is null
  // return <div>{user.name}</div>; // Runtime error if user is null

  // ‚úÖ Good: Check before using
  if (!user) {
    return <div>Loading user...</div>;
  }

  return <div>{user.name}</div>;
}

// Usage:
<UserCard user={null} /> // Shows "Loading user..."
<UserCard user={{ name: "Anna" }} /> // Shows "Anna"

4 Optional Properties in API Responses

APIs often return optional fields. Use ? to indicate a property may not exist. 

interface ApiResponse {
  name: string;
  age?: number; // May or may not be present
  phone?: string | null; // Can be undefined, null, or string
}

const response: ApiResponse = {
  name: "Anna",
  // age not present (undefined)
  phone: null // Explicitly null
};

console.log(response.age); // undefined
console.log(response.phone); // null

// In React:
function UserProfile({ data }: { data: ApiResponse }) {
  return (
    <div>
      <p>Name: {data.name}</p>
      <p>Age: {data.age ?? "Not specified"}</p>
      <p>Phone: {data.phone ?? "No phone"}</p>
    </div>
  );
}

5 Optional Chaining (?.) and Nullish Coalescing (??)

Two essential operators for working with null/undefined safely and cleanly.

Optional Chaining (?.): Access properties without crashing if they're null/undefined.

Nullish Coalescing (??): Provide default value only if null/undefined.

const user = {
  name: "Anna",
  address: undefined
};

// ‚ùå Without optional chaining
// const city = user.address.city; // Error: Cannot read property 'city' of undefined

// ‚úÖ With optional chaining
const city = user.address?.city; // undefined (no error)

// ‚ùå Without nullish coalescing
const port = process.env.PORT || 3000; // Problem: if PORT is 0 or "", uses 3000

// ‚úÖ With nullish coalescing
const safePort = process.env.PORT ?? 3000; // Only uses 3000 if PORT is null/undefined

// In React:
function UserCard({ user }: { user: User | null }) {
  const fullName = user?.name ?? "Unknown user";
  const email = user?.contact?.email ?? "No email";

  return <div>{fullName} - {email}</div>;
}

6 undefined vs null: When to Use Each in React

There's a subtle but important difference between using null and undefined in React.

React + TypeScript

// ‚úÖ Use null when you want to indicate "no value" explicitly
const [user, setUser] = useState<User | null>(null);

useEffect(() => {
  fetchUser().then(data => setUser(data || null));
}, []);

if (user === null) {
  return <div>Loading...</div>;
}

// ‚úÖ Use undefined for optional or missing values
interface Props {
  title?: string; // undefined if not passed
}

function Card({ title }: Props) {
  return <div>{title ?? "No title"}</div>;
}

// Comparison:
<Card /> // title is undefined
<Card title={undefined} /> // title is undefined (equivalent)
<Card title={null as any} /> // title is null (intentional)

7 Common Errors with null and undefined

The most frequent errors and how to avoid them:

‚ùå Error #1: Cannot read property of undefined

const user = undefined;
console.log(user.name); // ‚ùå Error: Cannot read property 'name' of undefined

// ‚úÖ Solution:
console.log(user?.name); // undefined (no error)

‚ùå Error #2: Object is possibly 'null'

function getUser(): User | null {
  return Math.random() > 0.5 ? { name: "Anna" } : null;
}

const user = getUser();
console.log(user.name); // ‚ùå Error: Object is possibly 'null'

// ‚úÖ Solution 1: Guard clause
if (!user) return;
console.log(user.name); // ‚úÖ OK (TypeScript knows it's not null)

// ‚úÖ Solution 2: Optional chaining
console.log(user?.name); // ‚úÖ OK

‚ùå Error #3: Using || instead of ??

const config = {
  port: 0, // Port 0 is valid
  debug: false
};

const port = config.port || 3000; // ‚ùå Returns 3000 (0 is falsy)
const safePort = config.port ?? 3000; // ‚úÖ Returns 0

const debug = config.debug || true; // ‚ùå Returns true (false is falsy)
const safeDebug = config.debug ?? true; // ‚úÖ Returns false

9 ‚úÖ Professional Best Practices

1. Always enable strictNullChecks

Protect your code from runtime errors. It's the first line of defense.

2. Prefer undefined for optional values

TypeScript uses undefined by default for optional properties. Follow this convention.

3. Use null for intentional absence

When you want to explicitly indicate "this value is empty", use null.

4. Use ?. and ?? for cleaner code

Optional chaining and nullish coalescing make your code more readable and safe.

5. Avoid ! (non-null assertion) in production

Only use ! if you're 100% sure. Prefer explicit validations.

6. In React, handle loading states explicitly

Differentiate between "loading", "empty", and "error" using null/undefined appropriately.
