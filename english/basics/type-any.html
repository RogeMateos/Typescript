Any Type

Problem

Sometimes I don't know what type a variable will have (API data, untyped libraries, migration from JS).

Solution

The any type disables type checking, allowing any value. It's a temporary "escape hatch".

‚ö†Ô∏è Warning

"Any is like a wildcard: it accepts everything, but you lose TypeScript's protection. Use it only when necessary!"

// Any accepts any type
let data: any = "text";
data = 123; // ‚úÖ OK
data = true; // ‚úÖ OK
data = { name: "Anna" }; // ‚úÖ OK

// Danger: no type checking
let x: any = "hello";
x.toFixed(); // ‚ùå Runtime error, but TS doesn't warn
// Better alternative: unknown (safer)
let data: unknown = "text";
data = 123; // ‚úÖ OK

// But you can't use it without checking first
// data.toFixed(); // ‚ùå Compilation error

if (typeof data === "number") {
  data.toFixed(); // ‚úÖ Now it's OK, because you verified
}

Best Practices

  • Avoid any whenever possible - it defeats TypeScript's purpose
  • Prefer unknown if you need flexibility with safety
  • Use any only temporarily in JS ‚Üí TS migrations
  • Enable noImplicitAny in tsconfig.json to avoid accidental any

Interview Prep

Q: When should you use the any type?

A: Only in specific cases: gradual JavaScript migration, integration with untyped libraries, or when the type is genuinely impossible to determine. In most cases, unknown is safer because it forces you to verify the type before using it.

Tutorial Objective

By the end of this tutorial you should:

  • Understand why any is dangerous
  • Know when to trust TypeScript's inference and when not to
  • Use casting correctly and with good judgment
  • Model real states with undefined
  • Write TypeScript that truly helps you in production

üí° Think of TypeScript as a safety net. This tutorial is about not cutting it.

1 The any type: what it is and why to avoid it

What is any

any disables TypeScript's type system for a variable.

let avenger: any = 123;
avenger = "Doctor Strange";
avenger = true;

TypeScript doesn't complain about anything.

The real problem (not theoretical)

When you use any:

You lose autocomplete
You lose early error detection
Bugs appear at runtime, not compile time
TypeScript stops adding value

Dangerous example:

let avenger: any = 123;

avenger = "Doctor Strange";
console.log(avenger.toFixed(2)); // üí• Runtime error

TypeScript didn't warn you. The bug reaches production.

Step-by-step explanation of the confusing example

Step 1: let avenger: any = 123;

What's happening here:

  • ‚Ä¢ You declare a variable called avenger
  • ‚Ä¢ You tell TypeScript: "this is any"
  • ‚Ä¢ That means: it can be anything and don't warn me about errors
  • ‚Ä¢ Right now: avenger === 123 (it's a number)

Step 2: avenger = "Doctor Strange";

What happens now:

  • ‚Ä¢ You change the value of avenger
  • ‚Ä¢ It's no longer a number, now it's a string
  • ‚Ä¢ But since the type is any:
  • ‚ùå TypeScript doesn't complain
  • ‚ùå TypeScript doesn't remember it was a number before
  • ‚Ä¢ For TS: "do whatever you want"

Step 3: console.log(avenger.toFixed(2));

Here's the problem üí•

  • ‚Ä¢ toFixed(2) only exists on numbers
  • ‚Ä¢ "Doctor Strange" does NOT have toFixed
  • ‚Ä¢ üëâ In JavaScript this is equivalent to:
"Doctor Strange".toFixed(2); // üí• error

Why didn't TypeScript warn?

Because you told it earlier:

let avenger: any;

That's like saying: "Don't protect me, I trust myself"

üí° The problem is NOT reassigning the variable

Reassigning is allowed in JavaScript and TypeScript.

The problem is:

  • ‚Ä¢ any allows mixing types without control
  • ‚Ä¢ TypeScript stops checking

üß† Simple analogy

Imagine any is taking off your seatbelt:

  • ‚Ä¢ While everything goes well ‚Üí nothing happens
  • ‚Ä¢ When there's a crash ‚Üí the damage is greater

TypeScript would normally tell you: "Hey, this isn't a number". But with any it stays silent.

‚úì How it should be done correctly

Option 1: use a real type

let avenger: number | string = 123;

avenger = "Doctor Strange";

if (typeof avenger === "number") {
  console.log(avenger.toFixed(2));
}

Now:

  • ‚Ä¢ TypeScript does protect you
  • ‚Ä¢ You only call toFixed when it's a number

Option 2: use unknown (better than any)

let avenger: unknown = 123;

avenger = "Doctor Strange";

if (typeof avenger === "number") {
  console.log(avenger.toFixed(2));
}

Here TypeScript forces you to think: "Are you sure this is a number?"

üîë Key idea to remember

  • ‚Ä¢ any allows silent errors
  • ‚Ä¢ unknown forces checking
  • ‚Ä¢ Real types prevent bugs

2 Type inference: when to trust it

Automatic inference

TypeScript is smart. You don't always need to write types explicitly:

let name = "Tony Stark";
// TypeScript knows: name is string

let age = 45;
// TypeScript knows: age is number

let hero = { name: "Iron Man", active: true };
// TypeScript knows: hero is { name: string, active: boolean }

When SHOULD you write the type explicitly?

When you want to enforce a restriction:

let role: "admin" | "user" = "admin"; // Can only be admin or user

In function parameters:

function greet(name: string) {
  console.log(`Hello, ${name}`);
}

For future values:

let userId: number; // Doesn't have a value yet
// ...
userId = getUserIdFromAPI();

3 Casting: when and how to use it

Casting is telling TypeScript: "Trust me, this value is of this type".

Casting syntax

let someValue: unknown = "Hello TypeScript";

// Option 1: 'as' syntax
let strLength1: number = (someValue as string).length;

// Option 2: angle-bracket syntax (doesn't work in JSX)
let strLength2: number = (<string>someValue).length;

‚ö†Ô∏è Warning: Casting does NOT convert values

It only changes how TypeScript sees the value, it doesn't transform it.

Dangerous casting

let data = "123" as any as number;
console.log(data.toFixed(2)); // üí• Error: "123" is still a string

Safe casting

let input: unknown = "123";

// First verify
if (typeof input === "string") {
  let num = parseInt(input); // Now it converts
  console.log(num.toFixed(2)); // ‚úÖ Works
}

üí° Golden rule

Use casting only when TypeScript can't infer correctly but you KNOW for certain the real type.

4 unknown vs any: the crucial difference

any

  • ‚Ä¢ Disables ALL checks
  • ‚Ä¢ You can do anything
  • ‚Ä¢ TypeScript doesn't help
  • ‚Ä¢ Errors at runtime

unknown

  • ‚Ä¢ Accepts any value
  • ‚Ä¢ But FORCES you to check before using
  • ‚Ä¢ TypeScript protects you
  • ‚Ä¢ Errors at compile time

Comparative example

// With any (DANGEROUS)
let data1: any = "hello";
data1.toFixed(); // ‚ö†Ô∏è TypeScript doesn't warn, fails at runtime

// With unknown (SAFE)
let data2: unknown = "hello";
data2.toFixed(); // ‚ùå Compilation error

// You have to check first
if (typeof data2 === "number") {
  data2.toFixed(); // ‚úÖ Now it works
}

‚úÖ Conclusion

If you need flexibility ‚Üí use unknown, not any

5 Uninitialized variables: the common mistake

Typical problem

let userId: number;

console.log(userId); // undefined (even though you say it's number)

‚ùå This is lying to TypeScript

You say userId is number, but it's actually undefined until you initialize it.

Correct solution

let userId: number | undefined;

console.log(userId); // undefined (expected)

// Check before using
if (userId !== undefined) {
  console.log(userId.toFixed(2)); // ‚úÖ Safe
}

Or better yet: always initialize

let userId: number = 0; // Default value
// or
let userId: number | null = null; // Explicitly without value yet

üí° Best practice

Model the real state: if something may not exist, use | undefined or | null

Complete example: Everything together 

// ‚ùå BAD: Dangerous code with any
function processData(data: any) {
  return data.toUpperCase() + " - " + data.toFixed(2);
}
processData(123); // üí• Fails at runtime

// ‚úÖ GOOD: Safe code with real types
function processDataSafe(data: string | number) {
  if (typeof data === "string") {
    return data.toUpperCase();
  } else {
    return data.toFixed(2);
  }
}
processDataSafe(123); // ‚úÖ Works: "123.00"
processDataSafe("hello"); // ‚úÖ Works: "HELLO"

// ‚úÖ GOOD: With unknown (when you don't know beforehand)
function processUnknown(data: unknown) {
  if (typeof data === "string") {
    return data.toUpperCase();
  } else if (typeof data === "number") {
    return data.toFixed(2);
  } else {
    return "Unsupported type";
  }
}

// ‚úÖ GOOD: Modeling real states
interface User {
  name: string;
  email: string | undefined; // May not have email
}

function sendEmail(user: User) {
  if (user.email === undefined) {
    console.log("User without email");
    return;
  }

  // Here TypeScript knows email exists
  console.log(`Sending to ${user.email}`);
}

Practical rules to remember

üö´ NEVER

  • ‚Ä¢ Use any out of laziness
  • ‚Ä¢ Cast without verifying
  • ‚Ä¢ Lie about types (e.g., let x: number; without initializing)

‚úÖ ALWAYS

  • ‚Ä¢ Prefer unknown over any
  • ‚Ä¢ Use real types (string | number)
  • ‚Ä¢ Check with typeof before using
  • ‚Ä¢ Model real states (| undefined if it may not exist)

üí° REMEMBER

  • ‚Ä¢ Casting does NOT convert, only changes how TS sees the value
  • ‚Ä¢ Inference works well 90% of the time
  • ‚Ä¢ TypeScript is your ally, not your enemy

üéØ If you only remember one thing: unknown > any
