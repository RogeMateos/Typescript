String Type

Problem

In JavaScript, you can assign any value to a variable that contained text, causing hard-to-track errors.

Solution

The string type in TypeScript ensures a variable only contains text, preventing type errors.

Visual Analogy

"A mailbox only accepts letters, not large packages. A string only accepts text, not numbers or objects."

// String declarations
let message: string = "Hello TypeScript";
let name: string = 'John'; // Single quotes also work
let multiline: string = `This is
a multiline
string`;

// message = 123; // ‚ùå Error: Type 'number' is not 'string'
// Template literals (backticks)
let name = "Anna";
let age = 25;

let greeting = `Hello, I'm ${name} and I'm ${age} years old`;
// Result: "Hello, I'm Anna and I'm 25 years old"

// Calculations inside templates
let price = `Total: ${10 + 5} euros`; // "Total: 15 euros"

Best Practices

  • Use template literals (`) to concatenate strings with variables
  • Prefer double quotes " for consistency (or single ', but pick one)
  • Avoid concatenating with +: use `Hello ${name}` instead of "Hello " + name

Interview Prep

Q: What is the difference between strings with quotes and backticks?

A: Backticks (`) allow creating template literals with variable interpolation using ${} and multiline strings. Single or double quotes only create literal strings without these features.

2 Ways to Define Strings in TypeScript

In TypeScript (and JavaScript) there are three main ways to define strings:

2.1 Single Quotes ' '

const hero: string = 'Batman';
  • üëâ Most commonly used by convention in many projects
  • üëâ Ideal when text doesn't contain internal single quotes

2.2 Double Quotes " "

const hero: string = "Green Lantern";
  • üëâ Work exactly the same as single quotes
  • üëâ Useful if text contains apostrophes (I'm Batman)

2.3 Backticks ` ` (Template Literals)

const hero: string = `Hero: Black Volcano`;

üëâ The most powerful:

  • ‚Ä¢ Allow interpolation
  • ‚Ä¢ Allow multiline
  • ‚Ä¢ Greatly improve readability

3 Template Literals: The Superpower of Strings

3.1 Variable Interpolation

const name = 'Batman';
console.log(`I'm ${name}`);
  • ‚úîÔ∏è Avoids unreadable concatenations
  • ‚úîÔ∏è Automatically converts numbers to string
const age = 42;
console.log(`Age: ${age}`);

‚ÑπÔ∏è This conversion is JavaScript, not TypeScript. TypeScript only validates types at compile time.

3.2 Multiline Strings

const message = `Hello,
this text
spans multiple lines`;

üëâ No , no concatenation, cleaner.

3.3 Interpolation with Logic

const isAdmin = true;
const role = `Role: ${isAdmin ? 'Admin' : 'User'}`;

4 Type Inference vs Explicit Typing

TypeScript infers types very well, especially with const.

4.1 Automatic Inference

const hero = 'Batman';
// TypeScript infers: string (or even the literal "Batman")
  • ‚úîÔ∏è Correct
  • ‚úîÔ∏è Recommended for simple values

4.2 Explicit Typing (when to use it)

Use : string when:

  • ‚Ä¢ The value can change (let)
  • ‚Ä¢ You're creating a public API
  • ‚Ä¢ There are union types
  • ‚Ä¢ You want explicit documentation
let hero: string = 'Batman';

üëâ Senior rule:

Type for safety and clarity, not out of habit.

6 String Manipulation and Safety

6.1 Index Access

const hero = 'Batman';
hero[0];   // 'B'
hero[10];  // undefined

‚ö†Ô∏è Out-of-range indices return undefined.

6.2 Optional Chaining (?.)

hero[10]?.toUpperCase();

üëâ If the value doesn't exist:

  • ‚Ä¢ Doesn't break the program
  • ‚Ä¢ Returns undefined

Analogy:

?. is a guardian that asks: "Does this exist before using it?"

7 Fallbacks: || vs ?? (very important)

7.1 OR Operator (||)

const result = '' || 'Fallback';

‚ùå Returns 'Fallback'

Because '' is falsy

7.2 Nullish Coalescing (??) ‚úÖ

const result = '' ?? 'Fallback';

‚úîÔ∏è Returns ''

Only replaces if null or undefined

7.3 Recommended Usage in Strings

console.log(hero[10]?.toUpperCase() ?? 'Not present');

üëâ Modern best practice in TypeScript

1 Basic component: strings + template literals 

import React from "react";

export function HeroIntro() {
  const batman = "Batman";
  const greenLantern = "Green Lantern";
  const blackVolcano = `Hero: Black Volcano`;

  const age = 53;

  return (
    <section>
      <h2>{`I'm ${batman}`}</h2>
      <p>{greenLantern}</p>
      <p>{blackVolcano}</p>

      {/* Interpolation with number (JS does String(age)) */}
      <p>{`Age: ${age}`}</p>
    </section>
  );
}

What this teaches (senior mode):

  • ‚Ä¢ In JSX, you can render strings directly or with template literals
  • ‚Ä¢ Interpolation is ideal for avoiding concatenations

2 Typed props: explicit typing where it adds value 

import React from "react";

type HeroCardProps = {
  name: string;
  universe?: string; // optional
};

export function HeroCard({ name, universe }: HeroCardProps) {
  return (
    <article>
      <h3>{name}</h3>

      {/* Optional chaining doesn't apply here because universe is string | undefined,
          we use fallback with ?? */}
      <p>{universe ?? "Unknown universe"}</p>
    </article>
  );
}

Why this is important:

  • ‚Ä¢ In React, typing provides a lot of value in props (component contract)
  • ‚Ä¢ ?? is better than || if you want to allow '' as a valid value

3 Real optional chaining with API data (nested object) 

import React from "react";

type User = {
  profile?: {
    name?: string;
  };
};

export function UserHeader() {
  const user: User = {}; // simulates incomplete API response

  return (
    <header>
      <h2>{user.profile?.name ?? "No name"}</h2>
    </header>
  );
}

Practical context:

  • ‚Ä¢ In React, this happens all the time with fetch/APIs
  • ‚Ä¢ ?. prevents errors like: "Cannot read properties of undefined"

4 State + string by index + safety 

import React, { useMemo, useState } from "react";

export function SafeCharViewer() {
  const [text, setText] = useState("Batman");
  const [index, setIndex] = useState(10);

  const result = useMemo(() => {
    return text[index]?.toUpperCase() ?? "Not present";
  }, [text, index]);

  return (
    <section>
      <h2>Safe character access</h2>

      <label>
        Text:
        <input value={text} onChange={(e) => setText(e.target.value)} />
      </label>

      <label>
        Index:
        <input
          type="number"
          value={index}
          onChange={(e) => setIndex(Number(e.target.value))}
        />
      </label>

      <p>{`Result: ${result}`}</p>
    </section>
  );
}

What you're teaching here:

  • ‚Ä¢ text[index] can be undefined
  • ‚Ä¢ ?. protects the .toUpperCase()
  • ‚Ä¢ ?? applies fallback only when the value is missing

5 Clear demonstration: || vs ?? in UI 

import React, { useState } from "react";

export function FallbackDemo() {
  const [value, setValue] = useState("");

  const withOr = value || "fallback with ||";
  const withNullish = value ?? "fallback with ??";

  return (
    <section>
      <h2>|| vs ??</h2>

      <input
        value={value}
        onChange={(e) => setValue(e.target.value)}
        placeholder="Leave empty to test"
      />

      <p>{`Current value: "${value}"`}</p>
      <p>{`With ||: "${withOr}"`}</p>
      <p>{`With ??: "${withNullish}"`}</p>
    </section>
  );
}

Senior explanation (simple):

  • ‚Ä¢ If the input is empty ('') ‚Üí || considers it "falsy" and uses fallback
  • ‚Ä¢ ?? keeps '' (because it's not null or undefined)
