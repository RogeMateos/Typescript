Any Type

Problem

Sometimes I don't know what type a variable will have (API data, untyped libraries, migration from JS).

Solution

The any type disables type checking, allowing any value. It's a temporary "escape hatch".

⚠️ Warning

"Any is like a wildcard: it accepts everything, but you lose TypeScript's protection. Use it only when necessary!"

// Any accepts any type
let data: any = "text";
data = 123; // ✅ OK
data = true; // ✅ OK
data = { name: "Anna" }; // ✅ OK

// Danger: no type checking
let x: any = "hello";
x.toFixed(); // ❌ Runtime error, but TS doesn't warn
// Better alternative: unknown (safer)
let data: unknown = "text";
data = 123; // ✅ OK

// But you can't use it without checking first
// data.toFixed(); // ❌ Compilation error

if (typeof data === "number") {
  data.toFixed(); // ✅ Now it's OK, because you verified
}

Best Practices

  • Avoid any whenever possible - it defeats TypeScript's purpose
  • Prefer unknown if you need flexibility with safety
  • Use any only temporarily in JS → TS migrations
  • Enable noImplicitAny in tsconfig.json to avoid accidental any

Interview Prep

Q: When should you use the any type?

A: Only in specific cases: gradual JavaScript migration, integration with untyped libraries, or when the type is genuinely impossible to determine. In most cases, unknown is safer because it forces you to verify the type before using it.
