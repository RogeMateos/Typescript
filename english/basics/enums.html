Enums: Fixed Values

Problem

Using magic strings like 'admin' or 'user' causes typos (e.g., 'admmin') that are hard to find.

Solution

Enums create a set of named constants, improving readability and preventing errors.

Visual Analogy

"Like a traffic light. It only has three possible states: Red, Yellow, or Green. No 'Blue'."

enum Role {
  Admin,
  User,
  Guest
}
const myRole: Role = Role.Admin;

Interview Question

Q: When should you use an Enum?

A: When you have a limited set of fixed values and want to avoid using loose strings or magic numbers.

üéØ What is an enum and why does it exist?

An enum in TypeScript serves to define a closed set of values with name and meaning.

üëâ The main goal is NOT to save code, but to:

  • Avoid magic values (0, 1, "max")
  • Improve readability
  • Improve autocomplete
  • Reduce human errors

‚ùå Code without enum:

audioSetting = 2;

‚úÖ Code with enum:

audioSetting = AudioLevel.Max;

Just by reading it, you already understand what's happening.

1 Basic enum definition (numeric enum) 

enum AudioLevel {
  Min,    // 0
  Medium, // 1
  Max     // 2
}

üìå TypeScript assigns numeric values automatically, starting from 0.

Usage:

let currentAudio: AudioLevel = AudioLevel.Medium;
console.log(currentAudio); // 1

2 How enums compile (important)

A numeric enum converts to a JavaScript object with bidirectional mapping:

AudioLevel[1] === "Medium"; // true

This generates extra code at runtime, something to keep in mind in frontend.

3 Enum with explicit values

You can control values manually:

enum AudioLevel {
  Min = 1,
  Medium, // 2
  Max = 10
}

‚ö†Ô∏è Caution: auto-increment continues from the last defined value.

4 String enums (recommended in frontend)

In React applications and APIs, string enums are the safest option.

enum AudioLevel {
  Min = "min",
  Medium = "medium",
  Max = "max"
}

Advantages:

  • More readable
  • No "phantom" numbers

Perfect for:

  • ‚Ä¢ APIs
  • ‚Ä¢ URLs
  • ‚Ä¢ localStorage
  • ‚Ä¢ logs
  • ‚Ä¢ UI states

Usage:

const level: AudioLevel = AudioLevel.Max; // "max"

5 Very important common error with numeric enums

This seems safe‚Ä¶ but it's not always:

enum AudioLevel { Min, Medium, Max }

function setAudio(level: AudioLevel) {}

setAudio(123); // ‚ö†Ô∏è can slip through in some contexts

üëâ Numeric enums behave like number in many places.

‚úÖ Recommended solutions:

  • Use string enums
  • Or use literal unions (as const)

6 Modern alternative (widely used in senior teams)

Many teams prefer this instead of enum:

export const AudioLevel = {
  Min: "min",
  Medium: "medium",
  Max: "max",
} as const;

export type AudioLevel =
  typeof AudioLevel[keyof typeof AudioLevel];
// "min" | "medium" | "max"
  • ‚úîÔ∏è Zero extra code at runtime
  • ‚úîÔ∏è Perfect typing
  • ‚úîÔ∏è Ideal for React

7 Using enums as type (the real power)

The true power of enums is here:

enum AudioLevel {
  Min = "min",
  Medium = "medium",
  Max = "max"
}

function setAudio(level: AudioLevel) {
  // logic
}

setAudio(AudioLevel.Max); // ‚úÖ

‚ùå Does not accept values outside the set.

8 Enums + safe logic (exhaustive switch)

This prevents future bugs:

function getLabel(level: AudioLevel): string {
  switch (level) {
    case AudioLevel.Min:
      return "Low";
    case AudioLevel.Medium:
      return "Medium";
    case AudioLevel.Max:
      return "High";
    default: {
      const _never: never = level;
      return _never;
    }
  }
}

üëâ If tomorrow you add a new value to the enum, TypeScript will force you to handle it.

Practical use in React

Example: UI state

enum ViewMode {
  Grid = "grid",
  List = "list"
}

function Products() {
  const [mode, setMode] = useState<ViewMode>(ViewMode.Grid);

  return (
    <>
      <button onClick={() => setMode(ViewMode.Grid)}>Grid</button>
      <button onClick={() => setMode(ViewMode.List)}>List</button>

      {mode === ViewMode.Grid ? <Grid /> : <List />}
    </>
  );
}

9 Mapping enums to configuration (clean pattern)

Avoid unnecessary if and switch statements:

const gainByLevel: Record<AudioLevel, number> = {
  [AudioLevel.Min]: 0.2,
  [AudioLevel.Medium]: 0.6,
  [AudioLevel.Max]: 1,
};

10 When NOT to use enums

Not everything needs an enum:

‚ùå Avoid enums if:

  • ‚Ä¢ You only need simple strings
  • ‚Ä¢ You want to minimize runtime JS
  • ‚Ä¢ You're modeling external data

üëâ In those cases, as const + union is usually better.

Mental checklist (this is senior level)

Before using an enum, ask yourself:

  • ‚Ä¢ Does this value leave my app? ‚Üí string enum / union
  • ‚Ä¢ Do I need reverse mapping? ‚Üí numeric enum
  • ‚Ä¢ Do I want zero runtime? ‚Üí as const
  • ‚Ä¢ Is it React + API? ‚Üí string enum or union

Final summary

  • ‚Ä¢ Enums create meaning, not just constants
  • ‚Ä¢ In frontend, string enums or unions are usually better
  • ‚Ä¢ Using them as types is where they really shine
  • ‚Ä¢ A good enum reduces bugs and makes code more expressive
